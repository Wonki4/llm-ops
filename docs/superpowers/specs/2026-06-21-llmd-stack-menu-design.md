# llm-d stack menu тАФ prefix-cache-aware serving via ArgoCD

**Date:** 2026-06-21
**Branch:** `feat/llmd-stack-menu` (new feature branch off `main`)

## Problem

The portal serves models by applying raw K8s manifests (Deployment/Service/
Ingress) directly through `K8sClient`. Operators want to stand up an
[llm-d](https://github.com/llm-d/llm-d) **precise-prefix-cache-aware** serving
stack for a chosen model тАФ a KV-cache-aware router (Gateway + EPP/inference
scheduler) in front of vLLM model servers that publish KV-block events тАФ to cut
TTFT. This needs a different deploy mechanism (Helm) and lifecycle than the
current raw-manifest path.

## Decisions

- **Deploy mechanism:** the portal generates an **ArgoCD `Application` CR**
  (GitOps). ArgoCD owns sync/drift; the portal only creates/patches/reads the
  Application and renders status.
- **Scope:** full-stack **new** deployment for a selected model (Gateway + EPP +
  llm-d-managed vLLM servers). Not fronting existing portal deployments.
- **Argo source:** an **internal Helm chart** (air-gap) referenced by the
  Application, with per-model **inline `helm.valuesObject`** generated by the
  portal. No external git repo.
- **Lifecycle:** create, **edit/scale**, delete, plus **live** status read.
- **Status:** read the Application CR back on each query (list/get). No
  reconciler / no DB-cached status in v1.

## Prerequisites (cluster-side, not portal-managed)

- ArgoCD installed; its namespace is configurable (`argocd_namespace`, default
  `argocd`).
- Gateway API + `gateway-api-inference-extension` CRDs installed.
- The internal Helm chart bundling the llm-d stack is published to an internal
  OCI/Helm registry; the llm-d / vLLM images are mirrored to an internal
  registry. A HuggingFace token Secret exists in the target namespace (referenced
  by name).

## Backend

- `app/services/llmd_manifests.py` (pure):
  - `build_llmd_values(stack, *, image_registry, hf_secret_name) -> dict` тАФ
    per-model helm values: model id, served name, replicas, gpu_count /
    resource_key, internal image registry, HF secret name, namespace.
  - `build_argo_application(stack, *, chart_repo, chart_version, values,
    argocd_namespace, destination_namespace) -> dict` тАФ the `argoproj.io/v1alpha1`
    `Application` CR: `spec.source` (repoURL=chart_repo, chart, targetRevision=
    chart_version, `helm.valuesObject`=values), `spec.destination`
    (server=in-cluster, namespace), `spec.syncPolicy.automated` (prune+selfHeal).
  - `argo_app_name_for(stack)` тАФ deterministic name.
- `app/api/llmd.py` тАФ super-user `/api/admin/llmd-stacks`:
  - `POST ""` тАФ create row, build Application, apply to `argocd_namespace` on the
    target cluster.
  - `GET ""` / `GET "/{id}"` тАФ list/get; **live**-read each stack's Application CR
    and merge `sync_status` / `health_status` / `status_message` into the
    response (best-effort; cluster-unreachable тЖТ status `Unknown`).
  - `PUT "/{id}"` тАФ edit/scale: update row, re-render values, **patch** the
    Application CR; ArgoCD re-syncs.
  - `DELETE "/{id}"` тАФ delete the Application CR (Argo prunes the workloads), then
    the row.
- `app/clients/k8s.py` тАФ add generic **CustomObjectsApi** support
  (create/patch/get/delete a namespaced custom resource) for the Argo
  `Application`. Existing typed helpers (Deployment/Service/Ingress/Job) are
  untouched. Cluster targeting reuses `cluster_id` / `services.clusters.
  k8s_for_cluster`.
- `app/config.py` тАФ `llmd_chart_repo`, `llmd_chart_version`, `llmd_image_registry`
  (air-gap internal defaults), `argocd_namespace` (default `argocd`),
  `llmd_hf_secret_name` (default e.g. `llm-d-hf-token`).
- Register the router in `main.py`.

## Frontend

- New admin page `admin/llmd` + a nav entry. List stacks (model, cluster,
  namespace, replicas) with **Synced/OutOfSync** + **Healthy/Progressing/
  Degraded** badges from the live status. Add/Edit dialog: model
  (`model_ref` + served name), cluster, namespace, replicas, gpu_count /
  resource_key, advanced raw-values override (optional). Delete with confirm.
- `use-api.ts` hooks (`useLlmdStacks`, create/update/delete), `LlmdStackSummary`
  type, i18n keys in `messages/en.json` + `messages/ko.json`.

## Testing

- Backend unit (`tests/test_llmd.py`):
  - `build_llmd_values` тАФ correct per-model values, internal registry, HF secret.
  - `build_argo_application` тАФ Application CR shape: source (repoURL / chart /
    targetRevision / valuesObject), destination, automated syncPolicy.
  - CRUD: create persists + builds; edit re-renders/patches; delete order.
- No live ArgoCD/GPU locally тЖТ end-to-end sync is out of scope here (same infra
  constraint as the multi-cluster work); verified at the manifest/CR level.

## Risks / out of scope

- v1 assumes ArgoCD + Gateway/inference CRDs + the internal chart already exist;
  the portal does not bootstrap them.
- Argo `destination` is the same (in-cluster) target in v1; cross-cluster Argo
  destinations are later work.
- No status reconciler тАФ if a stack's cluster is unreachable, status shows
  `Unknown` until the next query.
